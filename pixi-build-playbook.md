# Pixi 打包迁移手册

## 背景

本文总结了本项目将 Python、Node、Rust、Tauri 桌面打包统一收口到 `pixi.toml` 的成功经验，目标不是复述当前仓库的全部细节，而是提炼出一套可迁移到其他项目的做法。

适用场景：

- 新项目需要同时管理 Python、前端、Rust 或桌面壳工具链
- 现有项目想把“环境准备、开发启动、发布构建”统一到一套命令入口
- 需要降低“换一台机器就构建失败”“桌面子进程用错解释器”“网络下载不稳定导致打包失败”等问题

---

## 一、这套方案为什么有效

本项目的关键成功点，不是单纯“用了 Pixi”，而是把 **工具链版本、任务入口、构建脚本、外部依赖缓存、开发与发布的一致性** 一起设计了。

归纳下来，真正可复制的是下面五条原则：

### 1. 所有工具链由一个入口统一管理

在 `pixi.toml` 中统一声明 Python、Node、Rust、maturin 等依赖，让开发环境和构建环境尽量使用同一套版本来源，而不是依赖每位开发者本机各自安装的工具。

### 2. Pixi 负责暴露入口，复杂逻辑交给脚本

`pixi.toml` 中的任务保持简短稳定，例如：

- `bootstrap`
- `dev`
- `dev-backend`
- `build`
- `build-desktop`

而真正复杂的发布步骤，例如“先编译可选原生扩展，再构建前端，再准备嵌入式 Python，再调用 Tauri 打包”，交给 PowerShell 脚本处理。这样迁移时只需要保留任务名和流程骨架，再替换项目细节。

### 3. 开发模式与发布模式的关键命令要对齐

桌面开发模式中，Tauri 拉起的后端如果没有走 `pixi run dev-backend`，就可能意外使用系统 Python 或旧的 site-packages，导致“代码明明改了，但实际跑的不是当前仓库”。

本项目通过让开发脚本直接调用 Pixi 任务，而不是手写一套平行命令，避免了环境漂移。

### 4. 外部下载必须被显式管理

打包过程中最容易出问题的往往不是业务代码，而是外部资源：

- WiX 下载超时
- embeddable Python 压缩包损坏
- npm / pip 配置受开发者本机全局配置污染

本项目不是把这些问题留给“网络重试”，而是做成了可检查、可缓存、可重建、可文档化的独立步骤。

### 5. 文档与命令要一一对应

`docs/desktop-build.md` 和 wiki 中的 Pixi 相关命令，与仓库实际入口保持一致。这样迁移或交接时，不需要依赖“口口相传”。

## 二、本项目验证过的关键做法

### 1. 用 `pixi.toml` 收口环境与任务

本项目在 `pixi.toml` 中做了三件正确的事：

- 明确平台范围，如 `win-64`、`linux-64`
- 固定核心语言运行时版本，如 Python 3.12、Node 20
- 为开发、测试、构建、打包分别提供明确任务名

这样的好处是：

- 新机器初始化路径清晰：`pixi install` -> `pixi run bootstrap`
- CI 和本地机器的入口一致
- 迁移到新项目时，可以先复制任务结构，再替换内部命令

建议在新项目中至少保留以下任务分层：

- `install-python`
- `install-node-root`
- `install-node-web` 或其他前端子项目安装任务
- `bootstrap`
- `dev`
- `dev-backend`
- `build`
- `test`
- `typecheck`
- `build-desktop` 或等价发布任务

### 2. 依赖安装分层，而不是把所有命令塞进一个任务

本项目的 `bootstrap` 通过 `depends-on` 组合多个安装子任务，而不是写成一长串难维护的命令。

这是一个非常值得迁移的经验：

- 失败时更容易定位是 Python、根目录 npm，还是前端子目录 npm 出问题
- 后续如果增加第二个前端包或额外资源准备任务，不需要重写总入口
- 文档也更容易解释每一步在做什么

如果目标项目结构更复杂，可以继续沿用这个思路，把初始化拆成多个小任务，再用 `bootstrap` 统一收口。

### 3. 使用项目级 npm 配置，隔离本机环境污染

本项目用 `.npmrc.pixi` 配合 `--userconfig` 执行 npm 安装，这是一个很实用但常被忽略的细节。

它解决的问题是：

- 开发者自己的 `~/.npmrc` 里可能有私服、代理、镜像或兼容性设置
- 这些全局设置一旦影响构建，就会出现“某台机器能装，另一台机器失败”

迁移建议：

- 为 Pixi 任务准备一个项目级 npm 配置
- 所有由 Pixi 驱动的 npm 安装都明确指定 `--userconfig`
- 根目录与子项目都统一使用这套配置

这样能显著降低本机环境差异带来的不确定性。

### 4. `build-desktop` 只保留一个稳定入口

本项目没有把桌面打包暴露成多个互相独立、顺序靠人工记忆的命令，而是统一成：

```powershell
pixi run build-desktop
```

然后由脚本在内部按顺序完成：

1. 可选编译 Rust / PyO3 扩展
2. 前端 `npm ci` 与 `npm run build`
3. 准备嵌入式 Python 运行时
4. 预下载 WiX
5. 调用 Tauri 生成安装包

这套做法成功的关键点在于：

- 对使用者来说，入口足够简单
- 对维护者来说，流程仍然足够可控
- 任何一步失败都可以在脚本中做明确提示或提前退出

迁移时建议优先复制这种“单入口 + 脚本编排”的结构，而不是复制具体命令内容。

### 5. 可选增强能力允许降级，关键路径必须硬失败

本项目对构建步骤做了合理分级：

- 可选能力，例如 maturin 编译扩展，如果缺失或失败，可以回退到纯 Python 路径
- 关键能力，例如嵌入式 Python 运行时不完整、Tauri 打包失败，则必须终止

这种区分非常重要。否则常见的两种坏结果都会出现：

- 要么过于脆弱，一点可选能力失败就导致整个发布流程不可用
- 要么过于宽松，主链已经坏掉了却仍然输出“打包完成”

迁移建议：

- 先定义什么是“增强项”，什么是“发布必需项”
- 增强项可以告警并降级
- 必需项要检查关键文件、关键退出码、关键目录是否存在

### 6. 开发脚本必须复用 Pixi 中定义的后端命令

这是本项目最值得迁移的一条经验之一。

在桌面开发链路里，Tauri 的 `beforeDevCommand` 并没有直接写死 `python -m uvicorn ...`，而是通过脚本再调用 `pixi run dev-backend`。这样做的价值是：

- 保证桌面开发和手工启动后端使用的是同一条命令
- 避免 PATH 命中系统 Python 或旧依赖
- 减少“单独跑后端没问题，但 `tauri dev` 下行为不同”的隐性分叉

如果目标项目也有桌面壳、IDE 插件壳、Electron、Tauri 或类似宿主环境，建议都遵循这条原则：**宿主进程拉起的服务，也必须复用 Pixi 任务，而不是另写一套命令。**

### 7. 明确约束文件监视范围，避免错误热重载

本项目的后端开发命令显式指定了 `--app-dir` 与 `--reload-dir`，这是从实际踩坑中沉淀出来的经验。

如果监视范围过大，桌面开发过程中可能把构建产物、嵌入目录、第三方包目录也纳入热重载监控，导致：

- 频繁误重启
- 看似“代码一改就失控”
- 桌面开发体验非常不稳定

迁移建议：

- 热重载只监控真正需要监控的源码目录
- 不要默认监控整个仓库根目录
- 若有嵌入式运行时、bundle 目录、target 目录，应明确排除

### 8. 将嵌入式运行时准备做成独立脚本

本项目把嵌入式 Python 的准备单独做成 `stage-python-runtime.ps1`，而不是混在大构建脚本里直接写到一起。

这带来几个明显收益：

- 可以单独重建运行时
- 出问题时更容易复现与排查
- 可以在开发模式下复用它，用来模拟发布版运行环境

如果目标项目也需要随安装包携带运行时、模型、二进制工具、离线资源包，建议都采用同样方式：**把资源准备抽成可独立执行、可强制重建的单独脚本。**

### 9. 对外部下载做校验与缓存

本项目对 WiX 和 embeddable Python 的处理方式很值得保留：

- 有固定缓存目录
- 支持重新下载
- 下载后会做完整性检查
- 文档会告诉维护者如何手工清缓存再重试

这比“每次打包时临时在线拉资源”可靠很多。

迁移建议：

- 为每类外部资源约定缓存目录
- 如果资源有官方 checksum，下载后必须校验
- 如果资源经常损坏或超时，要提供单独的预下载脚本
- 文档中写明缓存位置、失败症状、手工恢复方式

### 10. 文档必须记录真实的打包顺序与排障入口

本项目的桌面构建文档不仅告诉你执行什么命令，还告诉你：

- 打包内部按什么顺序执行
- 最终产物在哪个目录
- 出问题看哪个日志文件
- 哪些网络依赖可能失败
- 哪些步骤只是开发模式下才有

这是“可交接”的关键。很多项目的问题不是打包链路不存在，而是只有原作者知道怎么排查。

迁移时建议让文档至少覆盖下面五项：

1. 初始化命令
2. 日常开发命令
3. 正式打包命令
4. 打包内部流程
5. 常见故障与日志位置

## 三、迁移到新项目时的推荐落地顺序

建议不要一开始就追求“完全复刻”。更稳妥的迁移顺序如下：

### 第一步：先统一工具链与任务名

优先完成：

- 新项目的 `pixi.toml`
- Python / Node / Rust 版本声明
- `bootstrap`、`dev`、`build`、`build-desktop` 等基础任务名

先让所有人都能通过 Pixi 跑通最基本的安装、开发、构建。

### 第二步：再迁移复杂脚本

把原项目中有价值的脚本设计迁移过去，例如：

- 统一打包入口脚本
- 外部资源预下载脚本
- 运行时准备脚本
- 桌面开发桥接脚本

此阶段重点不是“一字不差复制”，而是保留责任边界：

- Pixi 暴露入口
- 脚本负责编排
- 文档负责解释

### 第三步：最后补缓存、校验与故障文档

当主流程能跑通之后，再补强下面这些“让它稳定可交接”的能力：

- 校验下载资源
- 允许强制重建缓存
- 写清日志位置
- 写清常见网络问题与恢复方式

很多团队会止步于“我这里能打包”，但真正适合迁移的方案，必须做到“别人拿到仓库也能排障”。

## 四、迁移时最小可复制清单

如果你要把这套方案带到另一个项目，最少建议带走下面这些东西：

### 必带

- 一个明确的 `pixi.toml`
- 一组稳定的任务名：`bootstrap`、`dev`、`build`、`build-desktop`
- 一个统一的发布编排脚本
- 一个项目级 npm 配置
- 一份说明打包顺序与故障排查入口的文档

### 强烈建议带

- 独立的运行时准备脚本
- 外部资源预下载脚本
- 下载校验逻辑
- 对关键产物存在性的显式检查
- 宿主环境复用 `pixi run ...` 的开发启动链路

### 可按项目情况选择

- 可选原生扩展的降级策略
- 发布模式与开发模式的差异化行为
- 单独的缓存目录与离线恢复流程

## 五、常见风险与规避建议

### 风险 1：把 Pixi 只当成“装依赖工具”

如果只是把包版本放进 Pixi，但开发和打包仍然各跑各的脚本，最终还是会回到多入口、多环境、多套命令并存的问题。

规避建议：让 Pixi 成为对外唯一推荐入口，而不是可有可无的附加层。

### 风险 2：开发模式和发布模式各自维护一套后端启动命令

这会带来最难排查的问题：两个环境看似都能启动，但行为并不一致。

规避建议：让桌面开发、IDE 启动、手工调试都复用同一个 `dev-backend` 任务。

### 风险 3：把外部下载写进主流程，却没有缓存和校验

这样一旦网络波动，构建成功率会明显下降，而且问题难以复现。

规避建议：把下载逻辑抽出，显式缓存、校验，并允许手工预热。

### 风险 4：打包文档只写命令，不写内部顺序和日志位置

这会导致接手的人只能“反复重跑试试”，而不是有路径地排查。

规避建议：文档不仅要写怎么执行，还要写失败时先看哪里。

### 风险 5：热重载监控范围过大

桌面项目、嵌入式运行时项目、带构建产物目录的项目尤其容易踩这个坑。

规避建议：只监控源码目录，不监控整个仓库。

---

## 六、一句话结论

本项目最值得迁移的不是某一个具体命令，而是一套清晰的职责划分：

- `pixi.toml` 统一工具链与入口
- 脚本负责复杂编排
- 缓存与校验负责兜住外部资源不稳定
- 文档负责把真实流程与排障路径讲清楚

如果在新项目中把这四层一起迁过去，Pixi 才不仅是“换了个环境管理器”，而会真正变成一套稳定的开发与打包基础设施。
